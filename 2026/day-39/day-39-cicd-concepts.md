# Day 39 – What is CI/CD?

## Challenge Tasks

### Task 1: The Problem
Think about a team of 5 developers all pushing code to the same repo manually deploying to production.

Write in your notes:

1. What can go wrong?
  - Everything. Developers can accidentally overwrite each other's code. Manual file transfers (like FTP or SSH copy) are prone to human error (e.g., forgetting to upload a specific config file or forgetting to restart a service). There is no audit trail of who deployed what, making rollbacks a nightmare when the server inevitably crashes.

2. What does "it works on my machine" mean and why is it a real problem?
  - This is the classic developer excuse. It means the code runs perfectly on the developer's laptop (which might be a Mac with a specific version of Python and hidden environment variables) but fails entirely on the production server (which is Linux with different versions).

3. How many times a day can a team safely deploy manually?
  - Because manual deployments require intense coordination, downtime windows, and stress, teams might only deploy once a week or bi-weekly. It is a massive bottleneck.

### Task 2: CI vs CD
Research and write short definitions (2-3 lines each): Write one real-world example for each.

1. **Continuous Integration** — what happens, how often, what it catches
  - The practice of merging all developers' working copies into a shared mainline multiple times a day. Every merge triggers an automated build and test sequence. 
  - *Example:* A developer pushes a commit to GitHub. A pipeline automatically runs `npm test` or `pytest` to ensure no existing features were broken by the new code.

2. **Continuous Delivery** — how it's different from CI, what "delivery" means
  - An extension of CI. It ensures that the code is automatically built, tested, and packaged so it is *always ready* to be deployed to production. It stops just short of the live server, requiring a human to click an "Approve" button.
  - *Example:* Code passes testing and is automatically deployed to a "Staging" server. The QA team tests it, and the Release Manager clicks a button to send that exact same artifact to the live Production server.

3. **Continuous Deployment** — how it differs from Delivery, when teams use it
  - The absolute peak of automation. Every change that passes all stages of your production pipeline is released to your customers automatically, with zero human intervention.
  - *Example:* A Netflix engineer merges a new feature. The automated tests pass, the Docker image is built, and 15 minutes later, it is live for millions of users.

### Task 3: Pipeline Anatomy
A pipeline has these parts — write what each one does:

- **Trigger** — what starts the pipeline
    -  The event that starts the pipeline (e.g., a `git push`, opening a Pull Request, or a nightly cron schedule).

- **Stage** — a logical phase (build, test, deploy)
    - A major logical phase of the pipeline (e.g., the "Build" stage, the "Test" stage, the "Deploy" stage).

- **Job** — a unit of work inside a stage
    - A specific unit of work that executes on a single machine (e.g., "Run Backend Unit Tests"). Stages contain one or multiple jobs.

- **Step** — a single command or action inside a job
    - A single, individual command or action inside a job (e.g., running `npm install` or `docker build`).

- **Runner** — the machine that executes the job
    - The physical server or virtual machine that actually executes the jobs.

- **Artifact** — output produced by a job
    - A byproduct generated by a job that is saved for later use (e.g., a compiled `.jar` file, test coverage reports, or a Docker image).
